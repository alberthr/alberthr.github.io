## Count Distinct en Window Functions

Ni Databricks ni Oracle permeten escriure directament `COUNT(DISTINCT columna) OVER (PARTITION BY ...)`: el `DISTINCT` dins d'una funció de finestra no és vàlid a cap dels dos motors. El truc habitual és fer servir `DENSE_RANK()` sobre la columna d'interès: com que `DENSE_RANK` assigna el mateix número a valors repetits i no deixa buits, el valor màxim de `DENSE_RANK` dins d'un grup **és** el nombre de valors diferents.
=======
**Problema:**
L'estàndard SQL no permet l'ús directe de `COUNT(DISTINCT columna) OVER (PARTITION BY ...)` en molts motors de bases de dades degut a l'alta complexitat computacional que suposa gestionar l'estat dels valors únics dins d'una finestra mòbil.

**Solució:**
Es pot calcular el nombre d'elements únics dins d'una finestra combinant dues funcions `DENSE_RANK()`: una en ordre ascendent i una altra en ordre descendent. La suma d'ambdós rangs menys 1 equival exactament al nombre total de valors únics per a aquell grup.

**Implementació en SQL:**
Aquesta solució utilitza la sintaxi estàndard de Window Functions i és plenament compatible tant en Databricks SQL com en Oracle SQL:
>>>>>>> d868b1fb95186664b5ae09eaa2bcd77399beeaf7

```sql
-- Vàlid tant a Databricks com a Oracle
SELECT 
  client_id,
  categoria,
  DENSE_RANK() OVER (PARTITION BY client_id ORDER BY categoria ASC) + 
  DENSE_RANK() OVER (PARTITION BY client_id ORDER BY categoria DESC) - 1 
  AS total_categories_uniques
FROM transaccions;
```

`productes_diferents` dona, per a cada fila, el nombre de productes únics venuts a la seva botiga, sense necessitat de cap `GROUP BY` ni de perdre el detall de fila a fila.

## Filtrar sobre una Window Function

Filtrar pel resultat d'una funció de finestra (per exemple, quedar-se només amb la fila número 1 de cada grup) normalment obliga a embolcallar la consulta en una subconsulta només per poder aplicar el `WHERE` a fora, ja que les funcions de finestra no es poden fer servir directament al `WHERE`. Databricks SQL incorpora la clàusula `QUALIFY`, que ho evita:
=======
**Problema:** Les clàusules `WHERE` i `HAVING` s'avaluen abans que les funcions de finestra. Intentar filtrar el resultat d'un `ROW_NUMBER()`, `RANK()` o qualsevol mètrica de finestra en la mateixa consulta genera un error de sintaxi.

**Solució i Diferències de Sintaxi** 

- **Databricks SQL:** Disposa de la clàusula `QUALIFY`, la qual s'executa després de l'avaluació de les funcions de finestra, evitant la necessitat de crear CTEs o subconsultes.
- **Oracle SQL:** No implementa la clàusula `QUALIFY`. Cal fer servir una subconsulta o una CTE (*Common Table Expression*) per filtrar el resultat.

**Implementació en Databricks SQL**

```sql
SELECT 
  client_id,
  data_comanda,
  monto,
  ROW_NUMBER() OVER (PARTITION BY client_id ORDER BY data_comanda DESC) AS num_comanda
FROM comandes
QUALIFY ROW_NUMBER() OVER (PARTITION BY client_id ORDER BY data_comanda DESC) = 1;
```

## Maneig d'Arrays

**Problema:** En entorns de Big Data com Databricks (Spark SQL), sovint cal agrupar valors en estructures complexes tipus array per realitzar processament en un sol registre i, posteriorment, desfer aquesta estructura per recuperar la granularitat de files.

**Solució**
- `collect_set(columna)`: Agrega els valors únics d'un grup en un array.
- `collect_list(columna)`: Agrega tots els valors d'un grup en un array (mantenent duplicats).
- `EXPLODE(array)`: Descompon un array en múltiples files, generant una línia per cada element.

*Nota: Aquesta funcionalitat és específica de Databricks/Spark SQL i no té un equivalent directe d'un sol pas en la sintaxi tradicional d'Oracle SQL sense l'ús de tipus de dades objecte o XML/JSON.*

**Implementació en Databricks SQL**
>>>>>>> d868b1fb95186664b5ae09eaa2bcd77399beeaf7

```sql
-- Databricks: directe amb QUALIFY, sense subconsulta
SELECT botiga, producte, vendes
FROM vendes_diaries
QUALIFY ROW_NUMBER() OVER (PARTITION BY botiga ORDER BY vendes DESC) = 1;
```

## Desduplicar files

Un dels usos més pràctics de les window functions: quedar-se amb un únic registre per clau (per exemple, l'últim registre carregat per client) sense necessitat de `GROUP BY` amb agregacions complicades.

=======
**Problema:** Quan es calculen mitjanes o sumes mòbils mitjançant Window Functions, la definició del marc de la finestra (*window frame*) pot canviar el resultat si existeixen valors duplicats en l'eix d'ordenació (per exemple, diverses transaccions el mateix dia).

**Solució**

- `ROWS BETWEEN`: Compta exactament el nombre de files físiques especificades, independentment dels valors que continguin.
- `RANGE BETWEEN`: Inclou totes les files que caiguin dins del rang de valors lògics especificat. Si hi ha valors duplicats en la columna d'ordenació, tractarà totes les files amb el mateix valor com una sola entitat lògica.

**Implementació en SQL (Databricks i Oracle)**

>>>>>>> d868b1fb95186664b5ae09eaa2bcd77399beeaf7
```sql
-- Oracle (Databricks seria idèntic, o amb QUALIFY com a l'exemple anterior)
SELECT * 
FROM 
  (SELECT 
    client_id, data_actualitzacio, saldo,
    ROW_NUMBER() OVER (PARTITION BY client_id ORDER BY data_actualitzacio DESC) AS rn
  FROM historic_clients
)
WHERE rn = 1;
```

## Agrapació en Arrays

A Databricks, `collect_set(columna)` agrupa els valors únics d'una columna en un array, i `collect_list(columna)` fa el mateix però mantenint duplicats i l'ordre. Oracle no té una funció amb el mateix nom, però `JSON_ARRAYAGG` (des de la 12c) o `LISTAGG` amb `DISTINCT` cobreixen el mateix cas d'ús habitual.

=======
**Problema:** S'ha de realitzar el comptatge o la suma de diferents subconjunts de dades en una sola passada sobre la taula, evitant múltiples operacions `LEFT JOIN` o consultes dividides amb `UNION ALL`.

**Solució i Diferències de Sintaxi**

- **Sintaxi ANSI / Databricks SQL:** La clàusula `FILTER (WHERE ...)` s'afaga directament a la funció d'agregació. Expressa l'agregació condicional de forma més llegible i neta.
- **Oracle SQL / Sintaxi Universal:** Utilització de la funció `CASE WHEN` dins de la funció d'agregació.

**Implementació en Databricks SQL (`FILTER`)**

>>>>>>> d868b1fb95186664b5ae09eaa2bcd77399beeaf7
```sql
-- Databricks: array de productes únics per botiga
SELECT botiga, collect_set(producte) AS productes_unics
FROM vendes
GROUP BY botiga;

-- Oracle: equivalent amb JSON_ARRAYAGG (retorna un array JSON)
SELECT botiga, JSON_ARRAYAGG(DISTINCT producte) AS productes_unics
FROM vendes
GROUP BY botiga;
```

## Concatenar valors d'un grup

Per convertir els valors d'una columna, agrupats per una clau, en un sol text separat per comes:
=======
**Implementació Universal i Oracle SQL (`CASE WHEN`)**

```sql
-- Oracle
SELECT botiga, LISTAGG(producte, ', ') WITHIN GROUP (ORDER BY producte) AS llista_productes
FROM vendes
GROUP BY botiga;

-- Databricks
SELECT botiga, array_join(collect_list(producte), ', ') AS llista_productes
FROM vendes
GROUP BY botiga;
```


## `FILTER` en lloc de `CASE WHEN`

Quan cal calcular diverses sumes condicionals a la vegada (per exemple, vendes només dels caps de setmana, vendes només en promoció), la clàusula estàndard `FILTER` és més llegible que un `CASE WHEN` dins de cada `SUM`. Databricks la suporta; Oracle no, i cal el `CASE WHEN` clàssic.

```sql
-- Databricks: amb FILTER
SELECT botiga,
       SUM(vendes) FILTER (WHERE cap_de_setmana = TRUE) AS vendes_cap_setmana,
       SUM(vendes) FILTER (WHERE en_promocio = TRUE) AS vendes_promocio
FROM vendes_diaries
GROUP BY botiga;

-- Oracle: amb CASE WHEN (l'alternativa universal)
SELECT botiga,
       SUM(CASE WHEN cap_de_setmana = 1 THEN vendes END) AS vendes_cap_setmana,
       SUM(CASE WHEN en_promocio = 1 THEN vendes END) AS vendes_promocio
FROM vendes_diaries
GROUP BY botiga;
```


## Subtotals i totals amb `GROUPING SETS`, `ROLLUP` i `CUBE`

Per obtenir en una sola consulta el detall per botiga, el subtotal per regió i el total general (en lloc de tres consultes unides amb `UNION ALL`), `ROLLUP` calcula automàticament els subtotals jeràrquics:

```sql
-- Vàlid tant a Databricks com a Oracle
SELECT regio, botiga, SUM(vendes) AS total_vendes
FROM vendes
GROUP BY ROLLUP(regio, botiga);
```

`CUBE` fa el mateix però calculant totes les combinacions possibles de subtotals (no només la jerarquia regio→botiga, sinó també el total per botiga independentment de la regió), i `GROUPING SETS` permet especificar manualment exactament quines combinacions de subtotals es volen, sense generar-les totes.


## Mitjanes mòbils i totals acumulats amb `ROWS BETWEEN`

Les funcions de finestra permeten definir explícitament quantes files anteriors i posteriors s'inclouen en el càlcul, cosa que permet mitjanes mòbils i totals acumulats sense subconsultes ni auto-joins.

```sql
-- Vàlid tant a Databricks com a Oracle
SELECT botiga, data, vendes,
       AVG(vendes) OVER (
           PARTITION BY botiga ORDER BY data
           ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
       ) AS mitjana_movil_7dies,
       SUM(vendes) OVER (
           PARTITION BY botiga ORDER BY data
           ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS total_acumulat
FROM vendes_diaries;
```

## `PIVOT` i `UNPIVOT`

Convertir files en columnes (`PIVOT`) o columnes en files (`UNPIVOT`) directament en SQL, sense agregació condicional manual (`CASE WHEN` + `GROUP BY`) ni `UNION ALL`. Tant Databricks com Oracle tenen les dues clàusules natives, amb una diferència d'ordre a l'alias que sol confondre a `UNPIVOT`.

```sql
-- PIVOT a Oracle: 'valor' AS Alias
SELECT * FROM (
    SELECT botiga, mes, vendes FROM vendes_llarg
)
PIVOT (
    SUM(vendes) FOR mes IN ('Gener' AS Gener, 'Febrer' AS Febrer, 'Març' AS Marc)
);

-- PIVOT a Databricks: no cal l'AS
SELECT * FROM (
    SELECT botiga, mes, vendes FROM vendes_llarg
)
PIVOT (
    SUM(vendes) FOR mes IN ('Gener', 'Febrer', 'Març')
);

-- UNPIVOT a Oracle: Columna AS 'valor' (ordre invertit respecte a PIVOT)
SELECT botiga, mes, vendes
FROM vendes_ample
UNPIVOT (vendes FOR mes IN (Gener AS 'Gener', Febrer AS 'Febrer', Marc AS 'Març'));

-- UNPIVOT a Databricks
SELECT botiga, mes, vendes
FROM vendes_ample
UNPIVOT (vendes FOR mes IN (Gener, Febrer, Marc));
```
